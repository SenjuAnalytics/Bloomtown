## 1. NPC Architecture

### 1.1 Per-NPC component stack

Each NPC is a server entity with these components:

```
NpcEntity
├── NpcIdentity        (id, role, home, workplace)
├── NpcNeeds           (hunger, energy, social, ...)
├── NpcSchedule        (daily template + overrides)
├── NpcPersonality     (trait weights)
├── NpcRelationships   (map: entityId → RelationshipState)
├── NpcMemory          (recent events, gossip queue)
├── NpcInventory       (personal + role resources)
├── NpcStateMachine    (coarse life phase)
├── NpcUtilityBrain    (action scoring within phase)
├── NpcPathAgent       (navigation, current activity)
└── NpcSyncState       (client replication bundle)
```

### 1.2 Hybrid AI model

**State machine (coarse)** — answers *“What kind of moment is this NPC in?”*

| State | Entry | Exit |
|-------|--------|------|
| `Sleeping` | Schedule block + low energy | Wake time or interrupt |
| `Routine` | Default scheduled activity | Override wins or block ends |
| `NeedDriven` | Critical need crosses threshold | Need satisfied or timeout |
| `Social` | Conversation / group activity started | Activity ends |
| `Reacting` | Player action, event, danger | Reaction script completes |
| `Communal` | Joined village project | Shift ends or need interrupt |
| `Unavailable` | Story gate, injury, festival role | Designer/scripted unlock |

**Utility AI (fine)** — within `Routine`, `NeedDriven`, and `Social`, scores candidate actions every decision tick.

**Behavior trees (atomic)** — only for *how* to execute a chosen action (path to barn → play anim → deposit wheat). Keeps execution reliable; BT leaves are reusable across NPCs.

```
Every decision tick:
  1. State machine evaluates transitions (interrupts > schedule)
  2. If in Routine/NeedDriven/Social → UtilityBrain picks best action
  3. If action changed → Behavior tree runs until complete or preempted
  4. Needs decay; relationships/schedules update on events
```

### 1.3 Decision-making every tick

Not every NPC runs full AI every server frame.

**Tick tiers**

| Tier | Who | Rate | Work |
|------|-----|------|------|
| **Hot** | NPCs within player AOI (e.g. 64m), in conversation, in combat/event | Every 0.5–1 s | Full SM + utility + path |
| **Warm** | Same zone, outside AOI | Every 2–4 s | Needs decay + schedule check; utility if off-routine |
| **Cold** | Distant / unloaded slice | Every 30–60 s | Abstract simulation only |

**Abstract simulation (cold tier)**  
No pathfinding. Resolve schedule block → apply productivity/consumption formulas → update needs and inventory. On player approach, **hydrate** to full spatial state from abstract record.

## 4. Utility Scoring System

### 4.1 Formula

Each candidate action `a` gets:

```text
U(a) = Base(a)
     + NeedUrgency(a)
     + ScheduleFit(a)
     + PersonalityFit(a)
     + RelationshipFit(a)
     + EnvironmentFit(a)
     + MemoryFit(a)
     - Cost(a)
```

All terms are **floats** in roughly **−50..+80** range. Final scores compared after clamp to **0..150**.

**Selection rule:**

```text
winner = argmax U(a)
if U(winner) < MIN_THRESHOLD (40): use fallback ladder
if current action c exists and U(winner) < U(c) + HYSTERESIS (8): keep c
```

`HYSTERESIS` stops flicker (NPC oscillating between eat and work every 0.5s).

### 4.2 Term definitions

#### Base(a)
Designer-set per action type: `Eat=30`, `Sleep=35`, `Work=45`, `Chat=25`, `Flee=80`.

#### NeedUrgency(a)
For each need `n` that action satisfies:

```text
urgency(n) = (100 - value(n)) / 100          // 0..1
weight(n)  = roleNeedWeight[n] * criticalMult

criticalMult = 2.5 if n in Critical band, 1.5 if Low, 1.0 if Fine

NeedUrgency(a) = Σ satisfy[a,n] * urgency(n) * weight(n) * 40
```

`satisfy[a,n]` is 0–1 (eat → hunger 0.9, chat → social 0.6).

#### ScheduleFit(a)
```text
ScheduleFit(a) =
  +50  if a == current block default activity
  +30  if a == alternate allowed in same block (e.g. FreeTime)
  +15  if a == next block within 15 game-minutes (early lunch)
  -25  if a conf
```

## 6. Edge Cases & Conflict Resolution

### 6.1 Two critical needs at once

**Rule: lexicographic by survival, then by urgency gap.**

1. Compute `urgency(n) = (100 - value) / 100` for all Critical needs.  
2. Apply **survival order:** Health &gt; Energy &gt; Hunger &gt; Safety &gt; others.  
3. If top two within **0.05 urgency**, use role tie-break table:

| Role | Tie-break bias |
|------|----------------|
| Child | Safety &gt; Energy &gt; Hunger |
| Elder | Energy &gt; Health &gt; Hunger |
| Guard | Safety &gt; Energy |
| Default | Hunger &gt; Energy &gt; Social |

**Example:** Energy 12, Hunger 18 → both Critical; urgencies 0.88 vs 0.82 → **Energy wins** (survival order), NPC seeks bed not food. *Exception:* Hunger &lt; 5 → starvation override bumps Hunger first.

**Implementation:** `NeedDriven` state stores `primaryNeed` for BT routing; secondary need adds bonus to combined actions (eat at tavern also has benches → +Energy recovery).

### 6.2 Schedule vs player interaction

| Scenario | Resolution |
|----------|------------|
| Player talks during work | P4 pauses work; after dialogue, resume if block time &gt; 10 min |
| Player blocks path | Path fail → re-tick; utility may pick `WaitHere` or `RepathAround` |

### 6.4 Communal vs schedule vs need

Committed communal (P6) beats schedule. Critical need (P3) beats communal **only** if `need == Health` or `Energy < 10` (collapse risk). Otherwise NPC completes 15-min commitment slice then breaks for food — **honor partial obligation**.

### 6.5 BT failure loop

If action fails 3 times (path, target gone):

- Add `+100` penalty to that action for 5 minutes  
- Force fallback: schedule default at secondary location
